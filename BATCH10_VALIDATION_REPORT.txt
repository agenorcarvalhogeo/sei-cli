═══════════════════════════════════════════════════════════════
ROOT CAUSE DA REGRESSÃO NO PDF
═══════════════════════════════════════════════════════════════

O que acontece com o PDF (33920087, id_documento=41782477):

1. view_document_html() é chamado
2. _view_document_html_core busca link no arvore HTML → não encontra
3. Fallback: get_full_document_tree(expand_all=True) → encontra TreeDocument
4. Fetch TreeDocument.src_url (documento_visualizar) → retorna HTML
5. Mas para PDFs, o documento_visualizar retorna uma página wrapper
   com um <iframe> ou <embed> para o viewer de PDF — NÃO tem conteúdo
   textual do documento
6. parse_relatorio() recebe esse HTML wrapper → não encontra nada
7. Retorna relatorio com TUDO VAZIO
8. O código NÃO faz fallback para text_fallback porque
   view_document_html "succeeded" (retornou HTML, não exceção)

Antes do fix: view_document_html falhava com exceção →
código caía em text_fallback → extraía texto do PDF → resultados parciais.

Agora: view_document_html "sucede" com HTML do viewer →
parse_relatorio retorna empty → NÃO cai em text_fallback.
