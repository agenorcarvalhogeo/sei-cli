EMPTY_BODY_CHECK BUG REPORT — sei-cli
======================================
Date: 2026-03-30
Test document: 48774106 (Despacho 40372838, process 48756457)

SUMMARY
-------
empty_body_check returns false for newly created, unedited Despacho documents.
After investigation, the root cause is NOT a code bug in the check logic itself.
The issue is that the editable section (220) contains template boilerplate text
that is NOT effectively empty — it has real characters beyond &nbsp;.

ROOT CAUSE ANALYSIS
-------------------

Section 220 (editable=True) actual content after normalization:
  "Natal/RN, data da assinatura eletrônica."

This is template boilerplate injected by SEI into the editable section. It's not
user-written content, but it IS real text (not just &nbsp;), so
_is_effectively_empty_editor_content() correctly returns False.

The function chain works correctly:
  1. get_editor_sections() → 5 sections, section 220 has editable=True ✓
  2. quality-check isolates editable sections correctly ✓
  3. _normalize_editor_text() strips HTML, decodes entities → "Natal/RN, data da assinatura eletrônica." ✓
  4. _is_effectively_empty_editor_content() strips @template_vars@ → still has text → False ✓

SECTION MAP (document 48774106):
  316: editable=False, 129870 chars (header/timbre with base64 image)
  214: editable=False, 85 chars ("DESPACHO" title)
  217: editable=False, 444 chars (template: processo nº, interessado, @vars@)
  220: editable=True,  407 chars (body — has "Natal/RN, data da assinatura eletrônica.")
  317: editable=False, 638 chars (footer table)

THE FIX
-------
The empty_body_check needs to also strip KNOWN TEMPLATE BOILERPLATE from editable
sections before evaluating emptiness. Specifically:

In _is_effectively_empty_editor_content() (writing.py line 774), AFTER the existing
@template_var@ stripping (line 776), add stripping for standard boilerplate phrases:

```python
def _is_effectively_empty_editor_content(value: str) -> bool:
    normalized = _normalize_editor_text(value)
    # Strip @template_variables@
    normalized = re.sub(r"@[a-z0-9_]+@", " ", normalized, flags=re.IGNORECASE)
    # Strip known template boilerplate (SEI standard phrases)
    boilerplate_patterns = [
        r"Natal/RN\s*,?\s*data\s+da\s+assinatura\s+eletr[oô]nica\.?",
        r"data\s+da\s+assinatura\s+eletr[oô]nica\.?",
        r"Mossor[oó]/RN\s*,?\s*data\s+da\s+assinatura\s+eletr[oô]nica\.?",
        r"[A-ZÀ-Ú][a-zà-ú]+/[A-Z]{2}\s*,?\s*data\s+da\s+assinatura\s+eletr[oô]nica\.?",
    ]
    for pattern in boilerplate_patterns:
        normalized = re.sub(pattern, " ", normalized, flags=re.IGNORECASE)
    # Final cleanup
    normalized = re.sub(r"[^\wÀ-ÿ]+", "", normalized, flags=re.UNICODE)
    return not bool(normalized)
```

ALTERNATIVE (simpler, more generic):
Instead of pattern-matching specific boilerplate, use a GENERIC approach:
strip the boilerplate phrase "data da assinatura eletrônica" and city/state combos.
The simplest generic pattern that covers all SEI units:

```python
    # Strip "City/UF, data da assinatura eletrônica."
    normalized = re.sub(
        r"\w+/[A-Z]{2}\s*,?\s*data\s+da\s+assinatura\s+eletr[oô]nica\.?",
        " ", normalized, flags=re.IGNORECASE
    )
```

This single regex handles Natal/RN, Mossoró/RN, Brasília/DF, etc.
