BUG 1 — CRITICAL: Access level inheritance is fake
===================================================

SYMPTOM:
  sei document-create-preview 48756457 despacho --descricao "test" --json
  → access_policy.inherits_from_process = true
  → access_policy.nivel_codigo = "0" (público)

  sei document-create-confirm 48756457 despacho --descricao "test" --confirm --json
  → ok: false, error.code: "parse_error"
  → "SEI nao retornou id_documento apos a criacao"

EXPECTED:
  The process 48756457 is RESTRITO with hipótese "Informação Pessoal" (value=4).
  When --nivel is omitted, the CLI should either:
  (a) Detect the process access level and auto-fill nivel=1 + hipótese=4, OR
  (b) Fail BEFORE the POST with a clear error: "Processo é restrito. Use --nivel 1 --hipotese-acesso <id> --hipotese-campo selHipoteseLegal"

ACTUAL:
  inherits_from_process=true but nivel_codigo="0". The POST submits nivel=0
  to a restrito process. SEI reloads the form (missing required fields for
  restrito access). The CLI then fails to parse id_documento from the reloaded form.

ROOT CAUSE:
  The inheritance logic probably reads the form's default radio button state
  (which defaults to público/0) instead of querying the process's actual access level.

FIX OPTIONS (pick one):
  Option A (recommended): Before document-create, call process-open or inspect the
  process metadata to get its real nivel_acesso. If nivel > 0, auto-populate
  rdoNivelAcesso and fetch AJAX hypotheses. Set inherits_from_process=true only
  when the nivel was actually inherited.

  Option B (simpler): When --nivel is omitted AND the process is not público,
  raise a clear error requiring explicit --nivel. Set inherits_from_process=false
  and add a warning with the process's actual level.

WORKAROUND (current):
  Always pass --nivel 1 --hipotese-acesso 4 --hipotese-campo selHipoteseLegal
  explicitly. This works correctly (tested, doc 48761327 created successfully).

VERIFICATION TEST:
  # After fix, this should succeed:
  sei document-create-confirm 48756457 despacho \
    --descricao "Teste heranca" --confirm --json
  # Expected: ok=true, access_policy.nivel_codigo="1",
  #   access_policy.inherits_from_process=true (genuinely),
  #   access_policy.selected_hypothesis.value="4"


BUG 2 — MEDIUM: document-quality-check standalone doesn't detect editable sections
===================================================================================

SYMPTOM:
  sei document-quality-check 48761327 --process-id 48756457 --json
  → quality_check.editable_sections_count = 0
  → quality_check.edited_sections = []

  BUT document-edit-confirm (same document) returns:
  → quality_check.editable_sections_count = 1
  → quality_check.edited_sections = ["220"]

EXPECTED:
  Both paths should return identical editable_sections_count and edited_sections.

ROOT CAUSE:
  document-edit-confirm opens the editor (get_editor_sections) and parses CKEditor
  config to detect readOnly per section. document-quality-check standalone likely
  uses a different code path that does NOT open the editor or parse CKEditor config.

FIX:
  In the quality-check standalone path, call get_editor_sections() to load the
  editor page, then parse CKEditor readOnly config the same way edit-preview does.
  Extract editable sections and compare content against the template default to
  populate edited_sections.

  Alternatively, if opening the editor is too heavy for quality-check, at minimum
  document the limitation: "editable_sections_count requires editor access; standalone
  quality-check cannot determine editability."

VERIFICATION TEST:
  sei document-quality-check 48761327 --process-id 48756457 --json
  # Expected: editable_sections_count=1, edited_sections=["220"]
